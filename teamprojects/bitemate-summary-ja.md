# bitemate (Team-CAL) — インフラ・デプロイ担当 概要

## プロジェクト概要
店舗運営を支援する Web・モバイル統合プラットフォーム。シフト作成・勤怠・給与・書類管理に加え、**OpenCV (YOLO) による来店客数のカウント・混雑度算出**を備える。機械学習ライブラリを含む複数サービスのコンテナ統合が求められた。

- **リポジトリ:** https://github.com/kkmin20200331-max/Team-CAL-
- **技術スタック:** Spring Boot / React / FastAPI (OpenCV・YOLO) / Oracle Cloud ATP / Docker Compose / GitHub Actions / Nginx

## 担当範囲（関与の度合いで区分）
- **主導（自ら調査・設計・設定）:** Oracle ATP の接続構成およびトラブルシューティング (ORA-12529 / HikariCP)、`docker-compose.yml` によるコンテナ統合と環境変数 (`.env`) の分離設計
- **ペア作業:** GitHub Actions による CI/CD、各サービスの Dockerfile 作成・イメージ軽量化
- **調整のみ（主担当ではない）:** 既存 Nginx 設定を、コンテナが公開するポートへ転送先を合わせる調整。Nginx 自体の設計・SSL 構成の主担当ではない

## インフラ構成
```
ブラウザ (HTTPS/HTTP)
  └ Nginx（既存構成・リバースプロキシ / SSL 終端）
       ├ frontend コンテナ
       └ backend (Spring Boot)
            ├ opencv コンテナ (FastAPI / YOLO)  ← 客数・混雑度を内部連携
            └ Oracle Cloud ATP (TLS / Wallet)
```
外部ポートは HTTP 80 / HTTPS 443 のみ公開し、コンテナ間は共通ブリッジネットワークでサービス名により名前解決する構成とした。

## 主な成果 ― ORA-12529 の階層的な切り分け
接続時の **ORA-12529** は、複数の独立した原因が重なって発生していた。そのため手元の構成（`.env` / `docker-compose.yml` / VM 上の各 yml）から確認を始め、**ネットワーク (NSG) → Oracle 認証 (Wallet) → アクセス制御 (ACL)** へと外側に向かって階層的に切り分けた。

判明した複合原因は次の3点：
1. **ACL の反映漏れ** ― コンソールで IP を追加したが「保存」が未完了で、実際は拒否されていた
2. **Wallet の世代不一致** ― DB 再作成で更新された Wallet に対し、サーバー上に旧 Wallet が残っていた
3. **接続 URL の記述重複** ― `application.yml` と環境変数の指定が混在し、解釈に不整合が生じていた

解消後、起動時に**別レイヤー**の HikariCP "pool sealed" が発生。プールは初期化後に設定変更できない仕様のため、環境変数での上書きをやめ、Spring Boot の自動構成に委ねて解決した。

## 移行効果（VM 直接実行 → Docker Compose）
バージョン追跡（`sha-xxxxxxx` タグ）、実行環境の固定、サービス名による名前解決、コンテナ単位の自動再起動を実現し、再現性と運用安全性を高めた。

## 学んだこと
- **エラーを層別に切り分ける** ― 「接続失敗＝ネットワークだけ」と決めつけず、境界から段階的に固定して検証する
- **複合的な要因に粘り強く向き合う** ― 一つ直しても他の原因が残る前提でテストを繰り返す
- **担当の濃淡（主導／ペア／調整）を明確に区別する** ― チーム開発では自分の関与度を正確に把握することが重要
